# Анализ угроз и меры защиты

Документ соответствует разделам 3.1 и 3.2 ВКР — анализ потенциальных угроз и реализованные меры защиты.

## 3.1 Анализ потенциальных угроз

### XSS (Cross-Site Scripting)

**Угроза:** Внедрение вредоносного скрипта через пользовательский ввод (название книги, сообщение в обращении и т.д.), который выполняется в браузере других пользователей.

**Реализованные меры:**
- React по умолчанию экранирует все данные при рендере (`{variable}`), что предотвращает выполнение внедрённого HTML/JS
- Ввод пользователя не вставляется в `dangerouslySetInnerHTML`
- Серверная валидация и ограничение длины полей (subject ≤ 200, message ≤ 2000)

### SQL-инъекция (NoSQL-инъекция)

**Угроза:** Манипуляция запросами к базе данных через специальные символы в параметрах поиска или фильтрах.

**Реализованные меры:**
- Использование Mongoose ODM — все запросы параметризованы, пользовательский ввод не конкатенируется в строки запросов
- Поиск через `$regex` с экранированием спецсимволов (`escapeRegex`) — защита от ReDoS и неожиданного поведения
- Отсутствие `eval()` или динамической сборки запросов из пользовательского ввода

### Несанкционированный доступ (НСД)

**Угроза:** Доступ к данным или действиям без авторизации или с превышением прав (например, пользователь получает доступ к админ-панели).

**Реализованные меры:**
- JWT-токены в httpOnly cookie — токен недоступен из JavaScript, защита от XSS-кражи
- Middleware `protect` — проверка JWT на каждом защищённом запросе (cookie или Authorization)
- Middleware `checkRole("admin")` — разграничение доступа к админ-операциям (CRUD книг, обработка обращений)
- Проверка прав при чтении файла книги — только пользователь с активной выдачей может получить доступ к файлу

### Утечка паролей и XSS

**Угроза:** Пароли хранятся в открытом виде, JWT в localStorage уязвим к краже при XSS.

**Реализованные меры:**
- Хеширование паролей через bcrypt (соль, 10 раундов) — пароли не хранятся в открытом виде
- Поле `password` исключено из ответов API
- **JWT в httpOnly cookie** — токен недоступен из JavaScript, защита от кражи при XSS
- Рекомендуется использовать HTTPS при развёртывании

### Переполнение и DoS

**Угроза:** Большие объёмы данных или частые запросы могут перегрузить сервер.

**Реализованные меры:**
- Ограничение размера загружаемого файла (50 МБ) в Multer
- Валидация длины полей (имя ≤ 50, тема ≤ 200, сообщение ≤ 2000)
- Пагинация в каталоге книг (limit 1–100, по умолчанию 12)
- Rate limiting на `/api/auth` — 20 запросов за 15 минут (защита от брутфорса)
- Общий rate limiting на `/api` — 100 запросов за 15 минут (защита от DoS)
- Helmet — заголовки безопасности (X-Content-Type-Options, X-Frame-Options и др.)
- Whitelist полей в `createBook` и `updateBook` — защита от перезаписи служебных полей

## 3.2 Сводка мер защиты

| Угроза | Мера |
|--------|------|
| XSS | Экранирование в React, валидация длины |
| SQL/NoSQL-инъекция, ReDoS | Mongoose ODM, escapeRegex для $regex |
| НСД | JWT, middleware protect, checkRole |
| Утечка паролей | bcrypt, исключение password из ответов |
| DoS / переполнение | Лимиты файлов, валидация длины, rate limiting, пагинация |

## Дополнительные меры (надёжность)

- **Проверка env при старте** — приложение не запустится без MONGO_URI и JWT_SECRET
- **MongoDB retry** — до 3 попыток подключения с экспоненциальной задержкой
- **Глобальный error handler** — перехват необработанных ошибок, в production не раскрывает детали
- **404 handler** — единообразный ответ для несуществующих маршрутов

## Рекомендации при развёртывании

1. **HTTPS** — использовать TLS для шифрования трафика
2. **Переменные окружения** — JWT_SECRET, MONGO_URI хранить в `.env`, не коммитить
3. **CORS** — задать `CORS_ORIGINS` в `.env` (через запятую, например: `https://mysite.com,https://www.mysite.com`)
4. **NODE_ENV=production** — в production тестовые маршруты `/api/test` отключены
